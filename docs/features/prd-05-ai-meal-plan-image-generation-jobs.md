# PRD-05 — AI Meal Plan & Image Generation Jobs

## Feature Summary

Two chained Trigger.dev background jobs that produce a complete 7-day meal plan for a subscribed user. The first job (`generate-meal-plan`) calls OpenAI GPT-4o to generate structured meal data for 21 meals (3 per day × 7 days), persists each meal row to the `meals` table, then dispatches the second job. The second job (`generate-meal-images`) calls Replicate to generate a food photograph for each meal, uploads each image to Supabase Storage, records the file in the `files` table, and links it back to the meal row. When both jobs complete, the `meal_plans` row is updated to `status = 'ready'`. Progress is written throughout so the polling UI (PRD-06) can display a live percentage.

---

## User Problem

After submitting the generation form (PRD-04), the user has a `meal_plans` row in the database but no actual meal data. Nothing is displayed on the dashboard because `status = 'generating'`. The user is waiting and has no idea if anything is happening.

---

## User Goal

The user wants to receive a personalised 7-day meal plan — complete with meal names, nutritional info, and food imagery — without waiting on a slow synchronous request.

---

## MVP Scope

- Trigger.dev task `generate-meal-plan`: receives the job input payload from PRD-04, calls GPT-4o, inserts 21 `meals` rows, writes progress, then dispatches `generate-meal-images`.
- Trigger.dev task `generate-meal-images`: receives meal IDs, calls Replicate for each meal image, uploads to Supabase Storage, inserts a `files` row, links the file back to the meal, writes progress throughout.
- `meal_plans.status` set to `'ready'` when image generation completes, `'failed'` on unrecoverable error.
- `meals` table migration.
- DB helper functions for inserting meals, updating meal image links, and updating meal plan status.

---

## Out of Scope

- Any UI screens or polling logic (PRD-06).
- Regeneration or editing of individual meals.
- More than 3 meals per day (breakfast, lunch, dinner only).
- Nutritional targets beyond daily calories (macros are generated by the AI but not validated against targets).
- Caching or deduplication of AI-generated content.
- Webhook callbacks or server-sent events — status is polled (PRD-06).

---

## User Flow

1. PRD-04 server action creates a `meal_plans` row (`status = 'generating'`) and dispatches the `generate-meal-plan` job with the payload.
2. Trigger.dev picks up `generate-meal-plan`:
   a. Progress → 5%. Calls GPT-4o with user goal, daily calories, dietary preferences, and food category preferences.
   b. Parses the structured JSON response (21 meals).
   c. Inserts all 21 rows into `meals`. Progress → 40%.
   d. Dispatches `generate-meal-images` with the list of meal IDs.
   e. Progress → 50%. Calls `completeJob()`.
3. Trigger.dev picks up `generate-meal-images`:
   a. Iterates over the 21 meal IDs. For each meal:
      - Calls Replicate to generate a food photo.
      - Uploads the image to Supabase Storage at `meal-plans/<user_id>/<meal_plan_id>/<meal_id>.webp`.
      - Inserts a `files` row.
      - Updates `meals.image_file_id`.
      - Writes incremental progress (50–95% spread across 21 meals).
   b. Updates `meal_plans.status = 'ready'`. Progress → 100%. Calls `completeJob()`.
4. On any unrecoverable error in either job: updates `meal_plans.status = 'failed'`. Calls `failJob()`.

## Data Requirements

### New: `meals` table

```sql
create table public.meals (
  id              uuid primary key default gen_random_uuid(),
  meal_plan_id    uuid not null references public.meal_plans(id) on delete cascade,
  day             smallint not null check (day between 1 and 7),
  meal_type       text not null check (meal_type in ('breakfast', 'lunch', 'dinner')),
  name            text not null,
  description     text not null,
  ingredients     jsonb not null default '[]',
  calories        integer not null,
  protein_g       numeric(6,1) not null,
  carbs_g         numeric(6,1) not null,
  fat_g           numeric(6,1) not null,
  image_file_id   uuid references public.files(id) on delete set null,
  created_at      timestamptz not null default now()
);

-- Index for fetching all meals for a plan
create index meals_meal_plan_id_idx on public.meals(meal_plan_id);

-- RLS
alter table public.meals enable row level security;

create policy "Users can read their own meals"
  on public.meals for select
  using (
    exists (
      select 1 from public.meal_plans mp
      where mp.id = meals.meal_plan_id
        and mp.user_id = auth.uid()
    )
  );
```

## Business Logic

### Job 1: `generate-meal-plan`

**Task ID:** `generate-meal-plan`  
**Input payload (from PRD-04):**
```typescript
{
  meal_plan_id: string;
  user_id: string;
  daily_calories: number;
  goal: "lose" | "maintain" | "gain";
  dietary_preferences: string[];
  food_categories: string[];
}
```

**Steps:**
1. Call `updateJobProgress(jobId, 5)`.
2. Build a GPT-4o system prompt instructing it to return strict JSON — an array of 21 meal objects, one per combination of day (1–7) × meal_type ('breakfast'|'lunch'|'dinner'). Each object must include: `day`, `meal_type`, `name`, `description`, `ingredients` (array of strings), `calories` (integer), `protein_g`, `carbs_g`, `fat_g`.
3. Include `daily_calories`, `goal`, `dietary_preferences`, and `food_categories` in the user prompt so the AI can distribute calories appropriately and respect preferences.
4. Call OpenAI `chat.completions.create` with `model: "gpt-4o"`, `response_format: { type: "json_object" }`.
5. Parse the JSON response. Validate that 21 meal objects are returned. If parsing fails or count is wrong, call `failJob()`, update `meal_plans.status = 'failed'`, throw.
6. Call `updateJobProgress(jobId, 30)`.
7. Batch-insert all 21 rows into `meals` using `supabaseAdmin()`. Capture returned `id` values.
8. Call `updateJobProgress(jobId, 40)`.
9. Dispatch `generate-meal-images` job via `dispatchJob()` with payload `{ meal_plan_id, user_id, meal_ids: [<21 ids>], job_id: <new jobId from triggerJob> }`.
10. Call `updateJobProgress(jobId, 50)`.
11. Call `completeJob(jobId)`.

### Job 2: `generate-meal-images`

**Task ID:** `generate-meal-images`  
**Input payload:**
```typescript
{
  meal_plan_id: string;
  user_id: string;
  meal_ids: string[];  // exactly 21 IDs
}
```

**Steps (per meal, sequential to avoid Replicate rate limits):**
1. For each `meal_id` (index `i` of 21):
   a. Fetch the meal row to get `name` and `description`.
   b. Call Replicate with model `black-forest-labs/flux-schnell`, prompt: `"A professional overhead food photograph of {name}, {description}, on a clean white plate, natural lighting, restaurant quality"`.
   c. Download the image bytes from the Replicate output URL.
   d. Convert to WebP if not already (or upload as-is if already WebP/JPEG).
   e. Upload to Supabase Storage: `meal-plans/<user_id>/<meal_plan_id>/<meal_id>.webp` using `supabaseAdmin().storage.from('private-uploads').upload(...)`.
   f. Insert a `files` row via `supabaseAdmin()`: `{ owner_user_id: user_id, bucket: 'private-uploads', key: <storage path>, mime_type: 'image/webp', size_bytes: <bytes>, visibility: 'private', status: 'ready', metadata: { meal_id } }`.
   g. Update `meals.image_file_id = <new file id>` via `supabaseAdmin()`.
   h. Call `updateJobProgress(jobId, Math.round(50 + ((i + 1) / 21) * 45))` — progress runs from 50 → 95.
2. After all 21 meals: update `meal_plans.status = 'ready'` via `supabaseAdmin()`.
3. Call `updateJobProgress(jobId, 100)`.
4. Call `completeJob(jobId)`.

**Error handling:**
- If a single image generation fails, log the error and continue with remaining meals (partial images are acceptable for MVP).
- If the `meal_plans` update fails, log and re-throw (Trigger.dev will retry).
- If the job itself throws unrecoverably, call `failJob(jobId, error.message)` and update `meal_plans.status = 'failed'`.

### Calorie distribution guidance for GPT-4o prompt

| Meal type | Target share of `daily_calories` |
|-----------|----------------------------------|
| Breakfast | ~25% |
| Lunch | ~35% |
| Dinner | ~40% |

## Edge Cases

| Scenario | Handling |
|----------|----------|
| GPT-4o returns malformed JSON | Parse fails → `failJob()`, `meal_plans.status = 'failed'`, job stops |
| GPT-4o returns fewer than 21 meals | Validation fails → same as malformed JSON |
| Replicate times out for one meal | Log error, skip image for that meal, continue; `image_file_id` remains null |
| Replicate times out for all meals | All images skipped; plan still marked `'ready'` with null image IDs |
| Storage upload fails for one meal | Log error, skip file insert and meal update for that meal, continue |
| `meal_plans` status update fails at end of job 2 | Trigger.dev retries job 2; idempotent re-upload handled via overwrite on Storage upsert |
| Job 1 completes but job 2 never starts | `meal_plans.status` stays `'generating'`; user sees stuck progress; manual recovery via admin (out of scope for MVP) |
| User has no `dietary_preferences` or `food_categories` | Both default to empty arrays; GPT-4o prompt omits those constraints gracefully |
| `daily_calories` is very low (≤ 1200) | GPT-4o still distributes; no additional floor validation in the job |

## Acceptance Criteria

1. After the `generate-meal-plan` job runs successfully, exactly 21 rows exist in `meals` linked to the correct `meal_plan_id`, each with a non-null `name`, `description`, `calories`, `protein_g`, `carbs_g`, `fat_g`, and valid `day` (1–7) and `meal_type` values.
2. After the `generate-meal-images` job runs successfully, each meal row has a non-null `image_file_id`, a corresponding row exists in `files`, and the image binary is present in Supabase Storage at `meal-plans/<user_id>/<meal_plan_id>/<meal_id>.webp`.
3. `meal_plans.status` is `'ready'` after both jobs complete without unrecoverable error.
4. `meal_plans.status` is `'failed'` if the meal text generation step errors unrecoverably.
5. Job progress reaches 100 and `jobs.status = 'completed'` for both jobs on a successful run.
6. The jobs respect the user's `goal`, `dietary_preferences`, and `food_categories` — meals containing only allowed food types are generated (verified by inspection).
7. If `OPENAI_API_KEY` or `REPLICATE_API_TOKEN` are missing, the job fails cleanly with a descriptive error rather than an unhandled exception.
